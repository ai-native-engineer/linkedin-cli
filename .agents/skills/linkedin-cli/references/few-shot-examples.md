# 흥행글 말투 레퍼런스 (퓨샷)

링크드인 글을 새로 쓰기 전, 이 파일을 읽고 **말투(레지스터)와 문장 리듬**을 맞춘다.
여기 예시들은 실측 상위 코호트(한국어 LinkedIn 140건)에서 고른 실제 흥행글이다.

사용 규칙:
- **문장을 그대로 베끼지 않는다.** 가져오는 건 말투·호흡·구체성·훅 구조이지, 문장·소재가 아니다.
- 각 예시의 `말투 특징`이 왜 잘 읽히는지를 짚는다. 새 글이 이 결과 같은지 자가 점검한다.
- 작성자는 익명화했고(말투 학습엔 불필요), 지표(♥좋아요 💬댓글 🔁리포스트)는 2026-06-15 기준이다.
- 공통적으로 관찰되는 말투: **습니다 뼈대 + 거든요/근데/솔직히 구어체**, 격언·대구 만들기 회피,
  추상어 대신 **구체 고유명사·숫자**, 한 줄=한 문단 호흡, 스타카토 단정문.

## 예시 3 · 번호 가이드  (♥1416 💬13 🔁308)

**말투 특징:** 통념 대비 훅 + ❌/✅ 구체 예시. 단정 짧은 문단

```text
클로드 코드 잘 쓰는 사람과 못 쓰는 사람의 차이, 
저는 CLAUDE.md 하나라고 봅니다.
👉https://lnkd.in/gt7U_b-c

세션을 열 때마다 Claude는 기억을 초기화합니다. 
어제 했던 대화, 정해둔 규칙, 코딩 스타일. 전부 사라집니다.
그걸 해결하는 게 CLAUDE.md입니다.

CLAUDE.md를 제대로 쓰는 법, 12가지로 정리했습니다.
① CLAUDE.md = Claude에게 주는 지속적인 지침 파일
매 세션 시작 시 자동으로 로드됩니다. 
한 번 잘 써두면, 매번 설명할 필요가 없어집니다.

② 파일 위치가 범위를 결정합니다
관리 정책: /etc/claude-code/ → 조직 전체 강제 적용 
프로젝트: ./CLAUDE.md → Git으로 팀 공유 
사용자: ~/.claude/CLAUDE.md → 본인만, 모든 프로젝트
→ 더 구체적인 위치가 우선 적용됩니다.

③ 이렇게 쓰면 Claude가 무시합니다
❌ "코드를 제대로 포맷합니다" ✅ "2칸 들여쓰기 사용"
❌ "변경사항을 테스트합니다" ✅ "커밋 전 npm test 실행"
검증 가능할 만큼 구체적으로 쓰세요.

④ 파일당 200줄 이하를 지키세요
길수록 Claude의 준수율이 떨어집니다. 
내용이 많아지면 @import로 파일을 분리하거나 .
claude/rules/ 로 쪼개세요.

⑤ @import로 외부 파일을 불러올 수 있습니다
@README @package.json @~/.claude/my-preferences.md
Git에 올리기 싫은 개인 설정은 홈 디렉토리에서 불러오면 됩니다.

⑥ .claude/rules/ 로 규칙을 쪼개세요
code-style.md / testing.md / security.md 
주제별로 파일을 나누면 팀이 유지 관리하기 훨씬 쉬워집니다.

⑦ 경로별 규칙으로 컨텍스트를 아끼세요
paths: "src/api/**/*.ts"
이렇게 설정하면 해당 파일을 열 때만 규칙이 로드됩니다. 
항상 켜두지 않아도 됩니다.

⑧ 자동 메모리는 Claude가 스스로 쓰는 노트입니다
CLAUDE.md는 내가 씁니다. 자동 메모리는 Claude가 씁니다.
"항상 pnpm 써" → Claude가 알아서 기억합니다. 
세션이 끊겨도 다음에 기억하고 있습니다.

⑨ 로드 순서가 있습니다
관리 정책 → 사용자 → 상위 디렉토리 → 프로젝트 → 하위 디렉토리
모노레포에서 다른 팀 CLAUDE.md가 섞이면 
claudeMdExcludes 설정으로 제외할 수 있습니다.

⑩ /memory 명령을 습관으로 만드세요
현재 세션에 어떤 파일이 로드됐는지 바로 확인됩니다.
파일이 목록에 없으면 Claude는 볼 수 없습니다.
쓰고 나서 반드시 확인하세요.

⑪ /compact 후 지침이 사라졌다면
대화에만 있던 지침은 compact 후 소실됩니다. 
CLAUDE.md에 기록해두지 않으면 매번 다시 말해야 합니다.

⑫ 이것만 기억하세요
CLAUDE.md: 내가 쓰는 지침 — 구체적, 간결, 200줄 이하 
자동 메모리: Claude가 쓰는 노트 — 세션 간 자동 축적 
.claude/rules/: 대규모용 — 경로별 조건부 로드 
/memory: 로드 확인 필수 명령어

Claude Code를 제대로 세팅하는 방법, 
영상으로 자세히 정리했습니다.
👇아래 링크에서 확인하세요.
https://lnkd.in/gt7U_b-c
```

## 예시 4 · 자료배포·게이트  (♥480 💬797 🔁8)

**말투 특징:** 약자 전환 + 대괄호 게이트. 댓글 폭발(💬797)

```text

3월 21일 성수에서 스터디카페 사장님들 대상으로 클로드코드 강의를 했습니다.

참가자 전원, 코딩 경험 제로. (60대 분들도 무려 2분 계심)
ChatGPT도 "질문하고 복붙하는 게 전부"인 분들이었어요.

- 터미널이 무엇인지
- VSCode가 무엇인지 
(굳이 VsCode로 한 이유는 이후 바이브코딩에 대한 접근성이 더 좋게 하기 위해서 였습니다. 실제로 강의 끝나고 코딩 1도 모르는 분이 바로 바이브코딩으로 랜딩 페이지를 만드셨습니다.

차근차근 시작했습니다.

그리고 4시간 뒤.

이분들이 각자 본인 매장에 맞는
AI 포스터 디자이너를 직접 만들었습니다.

직원이 아니라 에이전트예요.
/designer 하고 엔터만 치면 매장 브랜드 컬러 맞춰서, 매장 정보 넣어서
포스터를 알아서 만들어주는 AI 직원.

━━━━━━━━━━━━━━━━━━━━━━━━━━

자영업자가 AI를 만나면 폭발력이 다른 것 같습니다.
혼자서 디자인, 마케팅, CS를 다 해야 하는 사람이니까요.한 명이 AI 팀을 갖는 순간, 변화가 바로 체감됩니다.

이번 강의의 흐름은 다음과 같았습니다.

► 터미널 열기 → 복붙 
► 클로드코드 설치 → 복붙
► 매장 정보 세팅 → AI에게 말하기
► AI 직원 만들기 → AI에게 말하기
► 포스터 제작 → AI에게 말하기

(단계별로 어떤 의미인지 원리 설명)

코드 한 줄 안 짜고 프롬프트로만 하였습니다.

근데 나온 결과물에 대한 수강생 반응:
"미리캔버스로 3시간 작업한 포스터보다
오늘 5분 작업한 포스터가 훨씬 이쁘네요"

━━━━━━━━━━━━━━━━━━━━━━━━━━

핵심은 간단합니다.

ChatGPT는 "대화"만 합니다.
물어보면 답하고, 복붙하는 게 끝.

클로드코드는 "실행"합니다.
파일을 직접 만들고, 수정하고, 저장합니다.
"말만 하는 직원"이 아니라 "직접 손으로 일하는 직원"입니다.

그리고 이 직원한테 매뉴얼을 한번 주면
다음부터는 세팅 없이 바로 일합니다.
새 이벤트 생기면 /designer 한마디면 끝.

에이전트를 하나씩 늘리면
디자이너, 블로그 작가, CS 매니저…
나만의 AI 팀이 만들어지는 겁니다.

저는 이 방법으로 스터디카페 2개 지점의운영 업무를 자동화하고 있고,지금은 이걸 가르치는 일도 하고 있습니다.
```

```text
피그마든 PPT든 결국 디자인의 수행 주체는 인간입니다. 도구는 인간의 손을 보조할 뿐입니다. 클로드 디자인은 수행 주체가 AI 본인입니다. 기획 단계에 약간의 인풋만 넣어줘도 디자인 시스템이 자동으로 잡히고, 세부 디자인까지 자기가 실행합니다. 제가 과거에 만든 프레젠테이션 몇 장을 학습시켰더니, 제가 미처 정의하지 못했던 디테일한 시스템 변수들까지 알아서 추출해서 잡아주더군요. 이 지점에서 가장 인상 깊었습니다.
```

```text
틀리지 않았다.

Tailwind도 정확히 컴파일했다. 

아무도 틀리지 않았는데 결과물이 달랐다.

추적하면서 크게 문제는 세 종류라는 걸 알게 됐다.

첫째, 오역. 값이 미묘하게 달라진다. (👀 눈에 보여서 그나마 잡을 수 있다.)

둘째, 브라우저 미지원 속성의 유입. 예를들어 Figma의 Vertical Trim은 CSS text-box-trim인데 Firefox가 아직 지원 안 한다. Figma의 자체 렌더링 엔진에서는 자연스럽게 동작하지만 브라우저 CSS 엔진에서는 안 되는 것들이다. 이건 예시 하나일 뿐이고 이런 간극이 곳곳에 있다. 

셋째가 제일 골치 아팠다. 조용한 누락이다.

Figma에는 있는데 CSS에 대응이 안 되는 것들이 있다. Figma의 stroke "Outside"는 CSS border로 재현할 수 없다. Constraints의 Scale/Center 모드는 CSS position: absolute의 고정 좌표 방식과 동작이 다르다. 블렌딩 모드 중 LINEAR_BURN, LINEAR_DODGE는 CSS mix-blend-mode에 대응 값이 없다. Figma의 자체 엔진에서는 되는데 CSS에서는 안 되는 것들이다.

이런 속성들이 MCP 블랙박스를 통과하며 최종적으로 Tailwind CSS로 번역될 때 매칭되는 클래스가 없거나 대응할 방법이 없으면 그냥 경고 없이 누락된다.
AI는 남은 속성으로 코드를 완성하고 "구현 완료"를 보고한다.

그래서 나름 고민해서 우회하는 워크플로우를 만들긴 했다.

디자인을 코드로 구현 전에 무조건 Figma를 먼저 Audit 한다. 
Claude Code가 REST API로 노드를 읽고 브라우저 미지원 속성이 설정된 프레임을 찾는다. 디자이너에게 교체를 요청한다. 가능한 경우

최초 컨텍스트가 문제가 있으면 가 다음단계도 줄줄이 땅콩이다.

구현할 때는 MCP와 REST API를 교차한다. 
MCP로 구조를 파악하고, /v1/files/nodes로 원본 수치를 직접 뽑는다. 
그나마 최소한의 블랙박스를 거친 원본에 가까운 수치라고 본다.

구현 후에는 헤드리스 브라우저로 검증한다. 
Claude Code가 getComputedStyle을 실행해서 브라우저 CSS 엔진이 최종 계산한 값을 읽는다. REST API 원본과 대조해서 불일치가 있으면 수정하고 다시 측정한다.

0. Figma 감사   → 미지원 속성 → 디자이너에게 교체 요청
1. REST API 추출  → 원본 수치
2. Claude Code 구현
3. getComputedStyle → 브라우저 실측값
4. 양 끝 비교   → 불일치 0 달성

솔직히 이 워크플로우도 효율적이지 않다. 그나마 효율적일 뿐
REST API로 노드 뽑고, 브라우저 띄워서 측정하고, 대조하고, 고치고, 다시 측정하는 루프는 느리다. MCP 하나로 끝나면 좋겠지만 자체 렌더링 엔진과 CSS 엔진이 다른 이상 안 된다.

그래도 현재 찾은 최선이다.

MCP 스크린샷 비교보다 getComputedStyle로 숫자 비교가 체감상 100배는 정확하다. 
Figma는 브라우저 안에서 돌아가지만 브라우저가 아니다.
웹은 결국 HTML과 CSS다. 

도구가 뭘 쓰든 최종 검증은 브라우저의 CSS 엔진이 계산한 값으로 해야 한다.
```
